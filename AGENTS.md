# AGENTS.md — Wikinest Автоскан

Инструкции для агентных coding-инструментов (OpenAI Codex CLI и аналоги) при работе в этом репозитории.

Это не дубль — это шлюз. Полные и актуальные правила живут в [CLAUDE.md](CLAUDE.md) (используется Claude Code для этого же репозитория). **Перед любой нетривиальной правкой прочитайте CLAUDE.md целиком** — здесь только самое критичное, на случай если до него почему-то не дойдёт очередь.

Оба файла должны совпадать в критичных пунктах ниже. Правите один — проверьте другой.

---

## Первое, что нужно понять: два независимых пути хостинга

- **`index.html`** — GitHub Pages, пишет через GitHub/GitLab Actions (`.github/workflows/`, `.gitlab-ci.yml`). Сейчас это **аварийный фолбэк**, включается вручную только если основная VM недоступна. Не трогать его API-слой (`ghFetch`, `dispatchApiWrite`, `watchDeploy` и т.п.), если явно не просят именно это.
- **`docs.html`** — тот же UI, но пишет через локальный бэкенд (`backend/`) на VM. Основной путь ежедневной работы.
- Бизнес-логика (рендеринг дерева, роли, поиск, редактор, модалки) — **должна быть идентична** в обоих файлах. API-слой (как читаем/пишем данные) — **разный и должен таким оставаться**. Не переносить транспорт одного пути в другой.
- Полная архитектура VM-пути и таблица отличий: [BACKEND_ARCHITECTURE.md](BACKEND_ARCHITECTURE.md). Деплой: [DEPLOYMENT.md](DEPLOYMENT.md).

## Жёсткие правила — ломают архитектуру или безопасность, если нарушить

- Тёмную тему не добавлять обратно ни в каком виде (удалена намеренно, была нерабочая).
- Пароли — только SHA-256 хеши в `roles.json`. Plaintext никогда не коммитить. `CREDENTIALS.md` — в `.gitignore`, не пушить.
- Динамические списки (поиск, избранное, недавние) — через `document.createElement` + `el.onclick = () => {...}`. Не через `innerHTML` с inline `onclick="...${JSON.stringify(x)}..."` — ломается, если в данных есть `"`.
- **В `docs.html` не переносить** паттерн «весь `roles.json` с хешами уходит в браузер» — там пароль проверяется на сервере (`POST /api/login`), `GET /api/roles` хеши не отдаёт вообще. Это не баг, это разный модели доверия для разных путей — см. [ACCESS_CONTROL.md](ACCESS_CONTROL.md).
- **В `docs.html`** `roles.json` меняется только через `POST /api/roles/visibility`, никогда через generic `put_file` — это блокируется сервером (`app.py`) намеренно, иначе легко потерять хеши паролей при записи с клиента, у которого их и так нет.
- **В `docs.html`** ссылки на ассеты (`/style.css`, `/config.json`, `/i18n/...`, `/docs/...`) — абсолютные пути, не относительные. Caddy отдаёт `docs.html` с любого URL через SPA-фоллбэк (`try_files`), относительные пути ломают загрузку при прямом заходе/reload на вложенной странице.
- **В `backend/Dockerfile`** `gunicorn --workers 1` — намеренно, не «забыли настроить». Сериализация записи (`write_lock` в `app.py`) и единственность фонового push-таймера в GitHub корректны только в одном процессе.
- **В `docker-compose.yml`** Caddy монтирует только явный список путей (`docs.html`, `style.css`, `config.json`, `i18n/`, `docs/`, `tree.json`, `search.json`) — не весь корень репозитория. `roles.json`, `.git/`, `backend/`, сам compose-файл не должны туда попасть ни при каком рефакторинге этого файла.
- `tree.json` на GitHub-пути пересобирает бот при пуше в `docs/**`, конфликт мержа гасится `--strategy-option=ours` (см. CLAUDE.md). На VM-пути его синхронно пересобирает бэкенд (`tree_builder.py`) при каждой записи — бота там нет и не будет. Не путать эти два механизма и не переносить один в другой.
- Загрузка медиафайлов (`docs.html`) — только через `POST /api/write/upload` (multipart), не через `put_file`/base64. Лимит 95MB — под жёсткий предел GitHub в 100MB на файл при пуше, см. [BACKEND_ARCHITECTURE.md](BACKEND_ARCHITECTURE.md#медиафайлы-картинки-видео).
