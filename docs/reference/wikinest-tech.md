## Краткое описание

У базы знаний два источника хранения: **Git** и **VM**. **VM** выставляется как приоритетная, при каждом изменении создает **git commit** и по автоматическому крону делает **git push** раз в час для синхронизации своих данных с **Git**. Второй же используется для хранения информации на случай отвала **VM** и как резервная ссылка на полностью живую и функционирующую базу знаний.
Все операции внутри **VM** происходят через **dispatch** эндпоинты, стандартные **CRUD** для работы с файлами документации + преобразователь и загрузка для документов и изображений для вставки оных в разделы документов. В **Git** так же реализован полный цикл работы с документами и папками, но реализация идет через **Github Actions**, которые вызываются через выстроенный токен в запросе для вызова. 

Предложение статьи работает аналогичным способом, но там добавляется **App-Pass** от конкретного провайдера почты и отправка происходит через стандартный **SMTP** через метод отправки.

Раз в год автоматический крон на ВМ создает новые base64 слепки ключей для всех ролей и отсылает их на: **pso@navi42.ru** и **zvv@navi42.ru**. Так же они сохраняются на сервере в отдельном документе. Если пароли будут утеряны, есть возможность сменить таргет почту на сервере и пересобрать их заново. Они автоматически будут загружены в репозиторий(закодированные) и использоваться в базе знаний сразу.

Если нет возможности взаимодействовать с корневым репозиторием - он доступен для зеркалирования на новом корпоративном аккаунте _(git аккаунт компании уже имеется)_ или же на совершенно новым с дубликатом базы на дату отрыва 1 час от последней актуальной синхронизации с сервером. **Гит репозиторий так же содержит полную копию бэкенда** _(читать инструкция по деплою)_ для того, чтоб всегда была на новой машине развернуть новую копию с полными возможностями и избежать полноценной потери доступа к базе знаний. Подобные действия сможет, при желании, сделать даже нейросеть, если потребуется, но **делалось это как вторая и независимая точка отказоустойчивости**, заложенная в архитектуру. 

По этой причине из соображений безопасности _(как и было сказано выше)_ были исключены все данные по **Ansible** и списку паролей для доступа к определенным ролям. Все это хранится и автоматически создается либо локально, либо на машине и не передается никуда, за исключением ежегодного диспатча-обновления паролей, которые уходят в обновленном виде на почту.   

Роутинг адресов доступа реализован следующим образом:


**Git** > https://keegooroomie.github.io/wikinest/

**VM** > https://95.190.201.194:26502/

Оба стараются резолвится до https://base.avtoscan42.ru по возможности, но приоритет отдается серверной, а не Git версии.

## Возможные проблемы

**VM недоступна**

→ использовать Git

**Git не синхронизируется**

→ проверить cron

**Не работают dispatch endpoint**

→ проверить backend

**Не создаются commit**

→ проверить права GitHub Token


По вопросам архитектуры, реализации или восстановления системы можно связаться с автором проекта через [GitHub][author]. В профиле размещены актуальные способы связи.
