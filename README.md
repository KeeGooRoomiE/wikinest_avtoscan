# WikiNest

**Git-native wiki for teams. Edit markdown in the browser, save as commits. No server, no build step, no dependencies beyond GitHub.**

## What it is

WikiNest turns a GitHub **or GitLab** repository into a fully functional wiki — with a split-view editor, sidebar navigation, full-text search, and password-protected editing — without requiring a server, a database, or a build pipeline.

Every page is a markdown file in `docs/`. Every edit is a commit. The sidebar tree is auto-generated by CI. The entire frontend is a single `index.html` + `style.css`.

## How it works

**GitHub**
```
Browser
  ├─ reads  ── raw.githubusercontent.com ── tree.json, .md files
  └─ writes ── api.github.com/repos/.../contents/...
                        └── commit → GitHub Actions → GitHub Pages
```

**GitLab**
```
Browser
  ├─ reads  ── gitlab.com/{owner}/{repo}/-/raw/main/... ── tree.json, .md files
  └─ writes ── gitlab.com/api/v4/projects/.../repository/files/...
                        └── commit → GitLab CI/CD → GitLab Pages
```

No backend. No database. Your PAT is encrypted with your password and stored in `localStorage` — it only ever reaches the provider API.

## Features

- **Split-view editor** — raw markdown on the left, live preview on the right; updates on every keystroke. Ctrl+S / ⌘S saves without leaving the keyboard
- **Page rename** — rename any page in-place from the toolbar; the tree reloads and navigates to the new path automatically
- **Anchor links** — TOC items update `location.hash` on click; deep links are shareable
- **Recently modified** — empty state shows the 5 most recently changed pages, sorted by last commit date
- **Unsaved changes warning** — browser confirms before closing a tab with pending edits
- **Tree navigation** — sidebar mirrors the `docs/` folder structure, unlimited nesting
- **Folder aliases** — custom display names via `_meta.json` without renaming directories
- **Full-text search** — client-side, ⌘K / Ctrl+K, searches titles and excerpts; results highlighted inline
- **Password-protected editing** — SHA-256 hash in `config.json`, PAT encrypted in localStorage
- **Multi-maintainer support** — `config.json` is the source of truth; each editor brings their own PAT
- **GitHub Actions status** — live deploy badge in the header; post-save watcher polls until deploy completes
- **i18n** — UI strings in `i18n/<lang>.json`; English and Russian included
- **Dark mode** — follows `prefers-color-scheme` automatically
- **Mobile-friendly** — sidebar collapses to hamburger, split-view collapses to editor-only
- **Zero dependencies** — one CDN script (`marked.js`), vanilla JS, no framework, no npm

## Quick start

### GitHub

**1. Fork and enable Pages**

Fork this repository. Go to **Settings → Pages → Source → GitHub Actions**. Go to **Settings → Actions → General → Workflow permissions → Read and write permissions**.

**2. Edit `config.json`**

```json
{
  "site_name": "My Wiki",
  "lang": "en",
  "provider": "github",
  "owner": "your-github-username",
  "repo": "wikinest",
  "edit_password_hash": ""
}
```

Commit and push. Actions will build and deploy in ~1 min.

**3. Create a PAT**

GitHub → Settings → Developer settings → Personal access tokens (classic). Required scope: `repo` ✓

---

### GitLab

**1. Fork and enable Pages**

Fork this repository on GitLab. Go to **Settings → CI/CD → Variables** and add a variable `GITLAB_TOKEN` with a PAT that has `api` + `write_repository` scope (used by the pipeline to commit `tree.json`).

GitLab Pages are enabled automatically once the `pages` job runs.

**2. Edit `config.json`**

```json
{
  "site_name": "My Wiki",
  "lang": "en",
  "provider": "gitlab",
  "owner": "your-gitlab-namespace",
  "repo": "wikinest",
  "edit_password_hash": ""
}
```

Commit and push. The pipeline will build and deploy in ~1 min.

**3. Create a PAT**

GitLab → User Settings → Access Tokens. Required scopes: `api` ✓

---

**4. Configure the wiki (both providers)**

Open your deployed site. Click **⚙**, enter your PAT and an edit password. Click Save.

## Configuration

`config.json` is committed to the repository and is public. Never put secrets here.

| Field | Type | Description |
|---|---|---|
| `site_name` | string | Displayed in the header and browser tab |
| `lang` | string | Default UI language (`en`, `ru`) |
| `provider` | string | `"github"` (default) or `"gitlab"` |
| `owner` | string | GitHub username / org, or GitLab namespace |
| `repo` | string | Repository name |
| `edit_password_hash` | string | SHA-256 hash of the edit password. Set via the ⚙ UI. |

`config.json` is loaded fresh on every page open — changes are picked up by all users without any local action.

## Authentication and security model

### Roles

| Role | What they need |
|---|---|
| **Reader** | Just open the site. No setup required. |
| **Editor** | Own PAT with `repo` scope + know the edit password |
| **Maintainer** | Write access to the repository (to change `config.json`, add pages via git) |

### How the edit password works

1. You enter a password in the ⚙ settings
2. SHA-256 hash is computed in the browser and stored in `config.json` via a commit
3. Your PAT is XOR-encrypted with that password and stored in `localStorage`
4. On each session, you enter the password → PAT is decrypted in memory → editing is available

### What is and isn't protected

| Protected | Not protected |
|---|---|
| PAT (encrypted in localStorage) | Page content (public GitHub repo) |
| Edit UI (password required) | `config.json` (public, readable by anyone) |
| Commits (require valid PAT) | Password hash (public, but SHA-256 is one-way) |

> WikiNest is designed for **internal or semi-public documentation** where content is not secret but editing should be controlled. Do not use it for confidential content in a public repository.

### Generating the hash manually

```js
// Run in browser console on any page
const pw = 'your-password';
const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(pw));
console.log(Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2,'0')).join(''));
```

## Creating and managing pages

**Create** — click **+ New page** in the sidebar. Enter a path:

| Input | Creates |
|---|---|
| `getting-started` | `docs/getting-started.md` |
| `ops/monitoring` | `docs/ops/monitoring.md` |
| `ops/servers/nginx` | `docs/ops/servers/nginx.md` |

Folders are created automatically. Page titles come from the first `# Heading` in the file.

**Rename** — open any page, click **Rename** in the toolbar. Enter the new path. WikiNest creates the file at the new location, deletes the old one, and navigates to the renamed page.

**Delete** — click **Delete** in the toolbar. A modal asks for confirmation before committing the deletion.

## Folder aliases

Add `_meta.json` inside any folder to set a display name:

```json
{ "title": "Operations" }
```

`_meta.json` files are never shown as pages in the sidebar.

## Internationalization

Add a language by:

1. Copying `i18n/en.json` to `i18n/<lang>.json` and translating all values
2. Adding `<option value="<lang>">XX</option>` to `#lang-switcher` in `index.html`

All keys must stay in English. Missing keys fall back to the key name.

## CI / CD

### GitHub Actions

Two workflows are included:

**`deploy.yml`** — triggers on every push to `main`. Rebuilds `tree.json`, commits it if changed, then deploys to GitHub Pages.

**`build-tree.yml`** — triggers only on changes to `docs/**`. Rebuilds `tree.json` (including `updated_at` timestamps) without re-triggering the full deploy.

Required: **Settings → Actions → General → Workflow permissions → Read and write permissions**

### GitLab CI/CD

**`.gitlab-ci.yml`** — two jobs:

- **`build-tree`** — triggers on `docs/**` changes on the default branch. Generates `tree.json` and commits it back using a `GITLAB_TOKEN` CI variable.
- **`pages`** — triggers on every push to the default branch. Copies all assets to `public/` and publishes to GitLab Pages.

Required: add a `GITLAB_TOKEN` CI/CD variable (**Settings → CI/CD → Variables**) with a PAT that has `api` + `write_repository` scope.

## Customization

All colors are CSS variables in `style.css`:

```css
:root {
  --accent:        #185FA5;   /* primary blue */
  --accent2:       #0c447c;   /* hover state */
  --accent-bg:     #E6F1FB;   /* accent tint backgrounds */
  --accent-border: #B5D4F4;   /* accent borders */
  --sidebar-w:     240px;
  --toc-w:         160px;
  --header-h:      48px;
}
```

Dark mode variables are in the `@media (prefers-color-scheme: dark)` block in the same file.

## Known limitations

- **Search is excerpt-only** — indexes only the first ~140 characters of each page. Increase `[:140]` in `build-tree.yml` to index more, at the cost of a larger `tree.json`.
- **No image upload UI** — images must be committed to the repository manually and referenced by path.
- **Single branch** — WikiNest always reads from and writes to `main`. Branch-based workflows are not supported.
- **No conflict resolution** — if two people edit the same page simultaneously, the second save will fail with a SHA mismatch error. Reload, re-apply changes, save again.
- **PAT rotation** — when a PAT expires, each user must re-enter their token in the ⚙ settings.
- **No offline support** — requires a live GitHub API connection.

## Contributing

PRs are welcome. For large changes, open an issue first to discuss.

No build step required. Open `index.html` directly in a browser for local development. For GitHub API calls to work, configure your own fork in the ⚙ settings.

- i18n: all new UI strings must have entries in both `en.json` and `ru.json`

### What to contribute

- Translations (`i18n/<lang>.json` + option in `index.html`)
- Bug fixes
- Accessibility improvements
- Documentation

## Troubleshooting

**Sidebar empty after creating, editing, or renaming a page**
GitHub Actions needs ~30–60 s to rebuild `tree.json`. The deploy status badge in the header tracks progress — wait for "Deployed" before reloading.

**"Could not load — check settings"**
Verify `owner` and `repo` in `config.json` are correct. Check that GitHub Pages is enabled and the first deploy completed successfully.

**Actions fail with "Permission denied"**
Settings → Actions → General → Workflow permissions → **Read and write permissions**.

**Password doesn't work after manually editing config.json**
Use the browser console method above to generate the exact SHA-256 hash.

**Save fails with "SHA mismatch"**
Someone else edited the same page simultaneously. Reload, copy your changes, and re-apply them.
