# Архитектура бэкенда (VM-путь)

> Второй, параллельный способ хостить эту же вики — рядом с GitHub Pages, не вместо неё.
> См. также: [PROJECT_ARCHITECTURE.md](PROJECT_ARCHITECTURE.md) (исходная, GitHub/GitLab CI архитектура), [DEPLOYMENT.md](DEPLOYMENT.md) (как раскатить).

---

## Зачем это существует

Компания выделила виртуалку. Вопрос был: держать на ней self-hosted GitLab + Runner и гонять те же CI-таски, что сейчас в `.github/workflows/`/`.gitlab-ci.yml` — или написать маленький бэкенд и убрать CI из пути записи вообще.

Выбор в пользу бэкенда: то, что сейчас делает CI на каждую запись (~170 строк питона на пересборку `tree.json`/`search.json` + пара HTTP-вызовов к Contents/Files API) — тривиальная нагрузка, а self-hosted GitLab — это Rails + PostgreSQL + Redis + Gitaly + Sidekiq ради неё же. Держать полноценный DevOps-продукт, чтобы по кнопке переписать markdown-файл, несоразмерно.

**Функционал для пользователя не меняется** — те же роли, тот же просмотр, то же редактирование. Меняется только то, что происходит между кликом «Сохранить» и файлом на диске.

## Два независимых пути

| | `index.html` (существующий) | `docs.html` (этот) |
|---|---|---|
| Хостинг | GitHub Pages | VM + Caddy |
| Хранилище | GitHub напрямую (Contents API) | локальный git checkout на VM |
| Запись | dispatch workflow → Actions → `secrets.GH_PAT` → commit | синхронный HTTP → файл на диск → commit локально |
| Скорость записи | секунды–минуты (CI round-trip) | доли секунды |
| Авторизация | пароль → SHA-256 → сравнение в браузере | пароль → бэкенд → серверная сессия (кука) |
| Роль сегодня | **аварийный фолбэк** — включается, если VM недоступна | **основной путь** ежедневной работы |

`index.html`, `roles.json`, `config.json`, `.github/workflows/`, `.gitlab-ci.yml` **не трогаются** — они продолжают работать как раньше и служат резервным путём записи, если VM недоступна.

### Дисциплина двух писателей

Раз оба пути технически умеют писать в контент, важно железное правило: **VM — единственный путь записи в штатном режиме.** `index.html`/GitHub Actions включаются только когда VM реально недоступна. Причина не в блокировке, а в физике: обычный `git push` без `--force` откажет, если удалённая ветка ушла вперёд локальной — то есть если во время аварии кто-то писал через GitHub, следующий автоматический пуш с VM просто перестанет проходить (см. `/api/sync/status` в [DEPLOYMENT.md](DEPLOYMENT.md#реконсиляция-после-аварийной-правки)), а не молча затрёт чужую правку. Но разбираться с этим руками всё равно придётся, поэтому аварийный путь — это осознанное исключение, не повседневный второй фронт.

## Хранилище: без базы данных

Данные — это по-прежнему `docs/*.md` + `tree.json` + `search.json` + `roles.json` + `config.json`, лежащие в git-репозитории. На VM это не отдельная копия — это тот же чекаут репозитория, что и весь код (`docker-compose.yml` монтирует корень репы в контейнер бэкенда как `/data`, см. [DEPLOYMENT.md](DEPLOYMENT.md)). Бэкенд — единственный процесс, которому разрешено писать в эту директорию.

Это специально не потребовало вводить БД: пересборка индекса (`backend/tree_builder.py`) — тот же алгоритм, что раньше жил в CI, просто вызывается функцией, а не отдельным процессом.

## Поток записи

```
POST /api/write/put_file {path, message, content}
  ↓
проверка сессии + прав на именно этот path (см. «Права» ниже)
  ↓
запись файла на диск (write_lock — один писатель за раз)
  ↓
если path начинается с docs/ → tree_builder.rebuild() пересобирает
  tree.json + search.json in-process (без похода в CI)
  ↓
git commit (только изменённые файлы, локально, без сети)
  ↓
200 OK
```

Никакого `sha` для optimistic locking, как в GitHub Contents API, не нужно — сериализация обеспечивается тем, что весь процесс работает через `write_lock` и один воркер gunicorn (см. `backend/Dockerfile`), а не сравнением ревизий.

### Что бэкенд починил по пути

`tree_builder.rebuild()` при пересборке сохраняет `roles[]`/`editors[]`/`isCounting` из предыдущего `tree.json` для каждой страницы. В CI-версии скрипт пересобирал `tree.json` с нуля на каждый пуш в `docs/**` и эти поля (они не выводятся из frontmatter, только через модалки «Видимость»/«Редакторы») тихо терялись при следующей правке любого файла. Сейчас это не воспроизведено — баг был латентным (ACCESS_CONTROL.md фиксирует, что `roles[]`/`editors[]` пока ни у одной страницы не задан), но раз функция переписывалась заново, исправить его было почти бесплатно.

`updated_at` берётся из `mtime` файла, а не из `git log -1` на файл (как было в CI-скрипте) — бэкенд сам пишет файл в момент правки, так что mtime и так точен, а форк `git log` на каждый файл при пересборке был не нужен.

## Медиафайлы (картинки, видео)

`POST /api/write/upload` — отдельный от `put_file` эндпоинт, `multipart/form-data`, а не base64-в-JSON. Для скриншота разница не критична, но `put_file`'ный паттерн (закодировать весь файл в base64-строку в памяти браузера, обернуть в JSON) для видео — плохая идея: +33% к размеру от base64, вся загрузка одним куском в памяти без стриминга. Werkzeug сам спулит входящий файл во временный файл при разборе multipart, `FileStorage.save()` — потоковая запись на диск, ни бэкенд, ни браузер не держат файл целиком в памяти как строку.

Загрузка не триггерит `tree_builder.rebuild()` — бинарный файл в `docs/assets/` не влияет на title/excerpt/структуру ни одной страницы, гонять весь `docs/` заново незачем. Коммитится тем же локальным `git commit`, что и всё остальное.

**Лимит 95MB** (`MAX_CONTENT_LENGTH` в `app.py`) — с запасом под жёсткий предел GitHub в 100MB на файл при пуше. Без лимита larger upload закоммитился бы локально без проблем и тихо ломал бы только ночной `git push` в GitHub — то есть страница из BACKEND_ARCHITECTURE.md «Бэкап в GitHub» перестала бы держать RPO ровно для этого файла, и вы узнали бы об этом через `/api/sync/status` постфактум, а не в момент загрузки. Превышение лимита — `413` сразу на запросе.

Если видео на регламентах вырастут за пределы разумного для гита (не разовые скринкасты, а систематически большие файлы) — следующий шаг это Git LFS или вынос `docs/assets/` из-под git вообще (файлы остаются на диске VM, читаются точно так же, просто не коммитятся и не пушатся — тогда бэкап для них нужен отдельный). Пока не делал ни то, ни другое — не было оснований усложнять под гипотетическую нагрузку.

## Авторизация

### Что изменилось относительно index.html

`index.html` целиком фетчит `roles.json` (**включая `password_hash` всех ролей**) в браузер и сравнивает SHA-256 на клиенте. Это нормально, пока единственный способ дотянуться до записи — токен с правом `actions:write`, спрятанный за GitHub-аутентификацией. На голом публичном URL с эндпоинтами это уже не ок: кто угодно может открыть страницу, вытащить все хеши и подбирать их офлайн, вообще не трогая UI.

На VM-пути:
- `POST /api/login` принимает пароль, сравнивает с `roles.json` **на сервере**, ставит подписанную HttpOnly-куку сессии (`SameSite=Strict`, `Secure` в проде).
- `GET /api/roles` отдаёт роли **без `password_hash`** — этого достаточно для UI (чекбоксы видимости разделов и т.п.), сами хеши браузер никогда не видит.
- Каждый `/api/write/*` и `/api/history*` проверяет роль из серверной сессии, а не из того, что прислал клиент.
- `/api/login` ограничен по попыткам (`auth.LoginRateLimiter` — 10 попыток / 5 минут на IP), иначе публичный эндпоинт логина — открытое приглашение перебирать пароль по сети.

### Права по операциям

Сервер воспроизводит ровно то, что сегодня решает видимость кнопок в `index.html` (не более и не менее):

| Операция | Правило | Где в текущем UI то же самое |
|---|---|---|
| `put_file` на существующую страницу `docs/*.md` | `can_edit` (owner) **или** роль в `page.editors[]` | `canEditPage()` |
| `put_file` на новый путь / `tree.json` | только `can_edit` | `createDoc`/`createFolder` гейтятся `requireUnlock()`→`canEdit()` |
| `put_file` на `roles.json` | **запрещено всегда** — только через `/api/roles/visibility` | см. ниже |
| `delete_file` | только `can_edit` | `btn-delete` виден только при `canEdit()` |
| `append_row` (глоссарий/инциденты) | роль ∈ `FULL_ACCESS_ROLES` (`owner`,`admin`,`director`,`head`) | `btn-append` виден по тому же условию |
| `/api/write/upload` (картинки, видео) | только `can_edit` | drag-drop гейтится `requireUnlock()`→`canEdit()` |
| `/api/history*` | только `can_edit` | `btn-history` виден только при `canEdit()` |

Один сознательный фикс: в оригинале `startEdit()` проверял только `canEdit()`, хотя кнопка редактирования показывается и при `canEditPage()` (т.е. если роль в `editors[]`) — то есть если бы `editors[]` где-то использовался, редактор для не-owner роли был бы виден, но не открывался. На VM-пути `startEdit()` проверяет `canEditPage()`, как и должен.

### `roles.json` — отдельный путь записи

Визуальные модалки «Видимость раздела» меняют `hidden_sections` конкретных ролей. В оригинале это шло как обычный `apiPutFile('roles.json', ..., полный JSON.stringify(roles))` — единственный способ это воспроизвести на VM было бы гонять `password_hash` туда-сюда через клиент, которого там больше нет. Вместо этого `POST /api/roles/visibility` принимает только `{folder_key, roles: {slug: bool}}`, применяет тот же тоггл, что раньше делал `saveSectionVisibility()` в JS, но **на сервере** — хеши паролей никогда не покидают `roles.json` на диске.

## Что не отдаётся наружу

`docker-compose.yml` монтирует в контейнер Caddy только явный список путей — `docs.html`, `style.css`, `config.json`, `i18n/`, `docs/`, `tree.json`, `search.json`. Всё остальное (`roles.json`, `.git/`, `backend/`, сам `docker-compose.yml`/`Caddyfile`, внутренние `*.md` вроде этого файла) физически не примонтировано в Caddy — их нельзя отдать по ошибке, потому что их там просто нет, а не потому что где-то стоит deny-правило, которое можно забыть.

## Бэкап в GitHub

- Каждая запись коммитится локально сразу (`git_ops.commit_paths`) — полная история по каждой правке сохраняется, как и сегодня.
- `git push` в GitHub идёт отдельным фоновым таймером (`PUSH_INTERVAL_SECONDS`, по умолчанию раз в сутки) — асинхронно, не блокирует ответ на запись.
- Максимальный разрыв данных при полной потере VM — это интервал пуша (сутки по умолчанию). Уменьшается бесплатно уменьшением `PUSH_INTERVAL_SECONDS`, потому что пуш ничего не стоит и не блокирует запись.
- `GET /api/sync/status` / кнопка «Запушить сейчас» в Настройках (`docs.html`) — на случай если пуш молча перестал проходить (истёк токен, сеть), это не всплывёт само, поэтому статус нужно где-то видеть, а не предполагать, что раз таймер тикает, значит бэкап живой.

## История версий

`GET /api/history?path=...` и `GET /api/history/content?path=...&sha=...` читают `git log`/`git show` из **локального** чекаута — тот же смысл, что «History» в `index.html` (там это `commits`/`contents?ref=` через GitHub API), только источник другой. Работает даже в день между пушами, потому что коммиты локальные и не зависят от того, что уже уехало на GitHub.

## Файлы

```
backend/
├── app.py            # Flask-роуты, сессии, права, фоновый push-луп
├── auth.py            # проверка пароля, sanitize roles, rate limiter, права
├── git_ops.py          # commit/push/log/show поверх локального git
├── tree_builder.py       # пересборка tree.json/search.json (порт CI-скрипта)
├── requirements.txt
└── Dockerfile           # gunicorn --workers 1 — см. комментарий в файле, почему именно 1
docker-compose.yml        # backend + caddy, bind-mount корня репы
Caddyfile             # /api/* → backend, всё остальное → статика + SPA-фоллбэк на docs.html
docs.html              # копия index.html с другим API-слоем (см. таблицу выше)
```
