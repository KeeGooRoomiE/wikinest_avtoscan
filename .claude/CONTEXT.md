# LLM Context — wikinest_avtoscan

## Текущее состояние проекта

WikiNest-база знаний компании «Автоскан». Два параллельных пути хостинга — см. CLAUDE.md «Два пути». Основной путь ежедневной работы — VM (`docs.html` + `backend/`, `/srv/wikinest` на сервере, SSH-алиас `wikinest-vm`). GitHub Pages (`index.html`) — аварийный фолбэк.

## Правило работы с контентом и кодом (с 2026-07-17)

**Все изменения — и код, и контент (docs/*.md) — теперь идут через: правка в локальном репозитории → коммит → `git push` → на VM `git pull` + restart нужных контейнеров.**

Раньше (до 2026-07-17) контент писался напрямую в git-репозиторий на VM через `docker compose exec backend python3 -c "tree_builder.rebuild(...); git_ops.commit_paths(...)"`, минуя локальный репозиторий. Это было быстро, но у ВМ-репозитория не было бэкапа на GitHub чаще раза в сутки — при восстановлении VM из снапшота часть несинхронизированной работы потерялась (см. ниже). Больше так не делать: ни для кода, ни для контента. `git_ops.commit_paths`/`tree_builder.rebuild` остаются внутренними примитивами бэкенда (их использует сам API при записи через UI), но Claude не должен вызывать их напрямую как обходной путь записи.

Исключение — `.env` на VM: гитигнорится (секреты — `SECRET_KEY`, `GIT_PUSH_TOKEN`), не версионируется. Правится напрямую через SSH, когда нужно.

## Пуш ВМ → GitHub (бэкап)

`PUSH_INTERVAL_SECONDS` в `backend/app.py` (`_push_loop`) — раз в сколько ВМ-репозиторий пушит себя в GitHub как бэкап (не часть пути записи, фоновый таймер). Было 86400 (сутки), с 2026-07-17 — **3600 (час)**: см. `docker-compose.yml` (дефолт) + `.env` на VM (явно переопределён, дефолт из compose там не работает, пока `.env` существует).

## Инцидент 2026-07-17: восстановление VM из снапшота

1С-разработчик делал точку восстановления сервера, сервер временно упал и был восстановлен из бэкапа на снапшот часовой-двухчасовой давности. Часть работы того дня (обновление таблиц CARVIS-камер и сопутствующего оборудования с миниатюрами) не долетела до бэкапа — восстановлена вручную из scratchpad и передеплоена. Всё остальное (карточки оборудования, все регламенты) на снапшоте уже было.

**Важный побочный эффект:** VM-репозиторий к этому моменту разошёлся с `origin/main` на ~470+ коммитов (весь массовый импорт архива документации в docs/devices, docs/processes) — они не пушились через push_loop (или push_loop ещё не срабатывал в первый час/сутки). Пока это не сверено, `git pull` на VM для НОВЫХ изменений может потребовать разрешения конфликта — проверить перед первым `git pull` после этой заметки, не полагаться слепо на fast-forward.

## Массовое наполнение документации (2026-07-16/17)

Из фотоархива клиента (`Карточки оборудования/`, `Регламенты/`) заполнены: все карточки оборудования под `docs/devices/*` (тахографы NRD/VDO/Штрих/Меркурий, навтерминалы Galileosky/Навтелеком, ДУТ Эскорт, Omnicomm, CARVIS видео, доп. оборудование, УОС, датчики нагрузки на ось) и регламенты под `docs/processes/{common,sales,service,support}` (~40 документов). Решение по регламентам: использовать существующую структуру `processes/{department}`, а не создавать отдельный `reglamentation/*` — там уже были процедурные инструкции по тем же 4 отделам.
