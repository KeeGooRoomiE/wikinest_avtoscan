## Что делаем и зачем

WikiNest превращается из простой wiki в полноценную базу знаний с ролевым доступом, структурированными разделами и улучшенным UI. Всё остаётся статической SPA без бэкенда — роли реализуются через пароли, хранящиеся в репе.

---

## Роли

| Роль | Slug | Видит | Может редактировать |
|---|---|---|---|
| Владелец | `owner` | Все разделы + все документы | ✅ Всё |
| Администратор | `admin` | Все разделы + все документы | ❌ Нет |
| Директор | `director` | Все разделы + все документы | ❌ Нет |
| Техническая поддержка | `support` | Все разделы кроме скрытых + подпапки ТП | ❌ Нет |
| Менеджер продаж | `sales` | Все разделы кроме Оборудования + подпапки менеджеров | ❌ Нет |
| Установщик | `installer` | TBD (уточнить позже) | ❌ Нет |

> Кнопки редактирования/создания отображаются ТОЛЬКО для `owner`.  
> "Общие" документы — без поля `roles` или `roles: []` — видят все.

## Фазы реализации

### Фаза 1 — Ролевая система
**Цель:** пароль → роль → сессия  
**Статус:** ✅ Готова (пароли не заполнены)

- [x] Создать `roles.json` — роли, хэши паролей, права (can_edit, hidden_sections)
- [x] Модальное окно входа (пароль → определение роли через SHA-256)
- [x] Роль сохраняется в `sessionStorage`
- [x] Функции `canEdit()`, `isHiddenPage()` для всего приложения
- [x] Бейдж с ролью + кнопка выхода в шапке
- [x] Кнопки редактирования видны только `owner`
- [x] Дерево фильтруется по `hidden_sections` роли

⚠️ **Нужно:** заполнить `password_hash` в `roles.json` для каждой роли.
Для генерации хэша — открыть консоль браузера и вызвать:
`generatePasswordHash('твой_пароль').then(h => console.log(h))`
