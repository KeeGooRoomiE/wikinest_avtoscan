# Контроль доступа — Wikinest Автоскан

Этот файл описывает все нестандартные условия по правам и видимости страниц.
Хранится только в репозитории, **не публикуется в документации**.

Актуальная модель — с 2026-07-21 (переработка ролей). Историю старой модели
(`hidden_sections`, FULL_ACCESS из 4 ролей, роли склад/логист/установщик)
см. в git-истории этого файла.

---

## Роли (roles.json)

| Роль (slug)      | Название               | can_edit | Особый статус |
|------------------|------------------------|:--------:|---------------|
| `owner`          | Владелец               | ✅       | Полные права на всё; **не отображается ни в одном чеклисте** видимости/редакторов — доступ неявный |
| `admin`          | Администратор          | ✅       | Полные права на всё; в чеклистах всегда виден — галочка стоит и **её нельзя снять** |
| `director`       | Директор               | ❌       | — |
| `head`           | Руководитель отдела    | ❌       | — |
| `support`        | Техническая поддержка  | ❌       | — |
| `sales`          | Отдел продаж           | ❌       | — |
| `accounting`     | Бухгалтерия            | ❌       | — |
| `office-manager` | Офис-менеджер          | ❌       | — |
| `service`        | Отдел сервиса          | ❌       | — |

Кроме `owner`/`admin` ни одна роль не имеет безусловного доступа — каждая
ограничивается на уровне конкретного документа или папки. Директор и
руководитель отдела — обычные ограничиваемые роли.

В коде: `FULL_ACCESS_ROLES = ['owner','admin']` (docs.html, index.html,
backend/auth.py — все три должны совпадать), `FORCED_ROLE = 'admin'`
(строка чеклиста с несъёмной галочкой).

## Принципы модели

1. **По умолчанию видно всем.** Пустой/отсутствующий `roles[]` у страницы или
   папки означает «без ограничений».
2. **Право редактирования включает видимость.** Роль из `editors[]` страницы
   видит её автоматически, даже если не входит в `roles[]`.
3. **Право редактирования включает управление доступом.** Кто может
   редактировать документ (`editors[]` или `can_edit`), тот может менять его
   видимость и список редакторов — кнопки «глаз»/«редакторы» завязаны на
   `canEditPage(page)`, а не на глобальное право.
4. **Ограничение папки каскадно.** `roles[]` папки действует на саму папку и
   всё её содержимое (страницы и подпапки). Побеждает самое специфичное
   (глубокое) правило по пути.

## Где что хранится

| Правило | Файл | Кто пишет |
|---------|------|-----------|
| Видимость страницы (`roles[]`), редакторы (`editors[]`), счётчик (`isCounting`) | запись страницы в `tree.json` | docs.html → `POST /api/write/page_visibility`; index.html → PUT всего tree.json через git-путь |
| Видимость папки (`roles[]`) | `_meta.json` самой папки | модалка «Видимость папки» (кнопка-глаз на заголовке папки в режиме редактирования дерева) |
| Роли и хеши паролей | `roles.json` | только вручную / `rotate-passwords` |

`tree_builder.py` переносит `roles[]`/`editors[]`/`isCounting` страниц из
предыдущего tree.json при каждой пересборке и создаёт `_folder`-маркер для
любой папки с непустым `roles[]` в `_meta.json` (иначе фронтенду неоткуда
это прочитать).

`hidden_sections` в roles.json больше не используется (очищен до `[]` у всех
ролей); поле пока живо в коде как legacy-проверка, но данных в нём нет.
Старая шестерёнка «Видимость раздела» удалена из UI.

## Право редактирования

`can_edit: true` (owner, admin) даёт:
- редактирование любой страницы, кнопку «Тестирование», черновики;
- структурные операции с деревом — создание/переименование/удаление папки,
  видимость папки, тумблер «Редактировать структуру» (реордер — только на
  VM-пути, см. [BACKEND_ARCHITECTURE.md](BACKEND_ARCHITECTURE.md#удаление-переименование-реордер-папок-2026-07));
- ручной Git Pull / Push в настройках (docs.html).

`editors[]` конкретной страницы даёт: редактирование этой страницы + управление
её видимостью/редакторами (`POST /api/write/page_visibility` проверяет
`can_edit_page` именно для этой страницы).

## Хранение паролей

Пароли хранятся как SHA-256 хеши в roles.json. Plaintext-пароли **никогда**
не коммитятся в git. Файл с паролями в открытом виде (CREDENTIALS.md) в
.gitignore и не пушится.

**У каждой роли пароль должен быть уникальным** — логин определяет роль по
хешу пароля, две роли с одинаковым паролем неразличимы (вторая станет
недостижимой). Это уже случалось с owner/admin.

Ежегодная ротация: `POST /api/admin/rotate-passwords` (только VM, сессия
owner/admin, ручной запуск) — перегенерирует пароли всех ролей и шлёт их
на настроенные почты. См. CREDENTIALS.md.

## Проверка прав — index.html vs docs.html

Всё выше описывает **логику** ролей (кто что видит/редактирует) — она общая для обоих путей. Разница в том, **где** это проверяется и это важно для того, что можно доверять клиенту.

**`index.html` (GitHub/GitLab Pages):** пароль сравнивается на клиенте против `roles.json`, который браузер получает целиком — включая `password_hash` всех ролей. Это приемлемо, потому что реальная запись идёт через workflow-токен, спрятанный за GitHub-аутентификацией — сам по себе клиентский пароль ничего не открывает напрямую.

**`docs.html` (VM):** эндпоинты бэкенда — голый публичный URL, поэтому здесь клиентская проверка ничего бы не защищала (обходится прямым запросом к API в обход интерфейса). Пароль сверяется на сервере (`POST /api/login`), сессия — подписанная HttpOnly-кука, `GET /api/roles` не отдаёт `password_hash` вообще. Права (`can_edit`, `editors[]`, `FULL_ACCESS_ROLES` для append_row) проверяются на каждый write-запрос сервером из своей копии `roles.json`/`tree.json`, а не из того, что прислал клиент. Подробности и таблица прав по операциям — [BACKEND_ARCHITECTURE.md](BACKEND_ARCHITECTURE.md#авторизация).

Не переносить клиентскую SHA-256-проверку из `index.html` в `docs.html` — там это дыра, а не эквивалентная замена.

**Важное отличие видимости от защиты:** скрытие страниц/папок по `roles[]` на
обоих путях — клиентская фильтрация дерева. На GitHub-пути содержимое в любом
случае доступно любому, кто откроет raw-файлы репозитория; на VM-пути
`GET /docs/*.md` тоже публичен. `roles[]` — это управление *отображением*
для сотрудников, а не криптографическая защита содержимого. Действительно
секретное в базу знаний не класть.
