# Деплой на VM

Runbook для варианта из [BACKEND_ARCHITECTURE.md](BACKEND_ARCHITECTURE.md). Весь деплой — это клонировать репозиторий и поднять два контейнера; `docs.html`, `backend/`, `docker-compose.yml`, `Caddyfile` уже лежат в репе, отдельно раскатывать нечего.

---

## Предварительно

- VM с Docker + Docker Compose plugin.
- Домен под контролем: сейчас `CNAME` в репозитории — `base.avtoscan42.ru`. Он приписан к GitHub Pages; см. «Домен» ниже про то, что с ним делать.
- GitHub-репозиторий: `KeeGooRoomiE/wikinest_avtoscan` (из `config.json`).
- **Если VM за NAT провайдера с расшаренным IP (как сейчас)** — снаружи реально доступен только тот набор портов, что провайдер явно пробросил на вашу машину, а не стандартные 80/443 (те могут принадлежать чужому сервису на том же IP — см. «Домен» ниже). Проверить до деплоя: `curl -v http://<публичный IP>/` — если отвечает не ваш Caddy, а что-то незнакомое (нашёлся, например, чужой nginx/1С-панель), стандартные порты не ваши, идём по ветке «порт вместо домена».

## 1. Клонировать репозиторий на VM

```bash
git clone https://github.com/KeeGooRoomiE/wikinest_avtoscan.git /srv/wikinest
cd /srv/wikinest
```

Этот чекаут и есть данные — `docker-compose.yml` монтирует его же в контейнер бэкенда. Отдельной директории под контент заводить не нужно.

## 2. Секреты — `.env`

`.env` в `.gitignore`, в репозиторий не попадает — создаётся один раз на VM.

```bash
SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")
cat > .env <<EOF
SECRET_KEY=${SECRET_KEY}
GIT_PUSH_REPO=KeeGooRoomiE/wikinest_avtoscan
GIT_PUSH_TOKEN=
PUSH_INTERVAL_SECONDS=86400
EOF
```

- **`SECRET_KEY`** — подписывает сессионную куку. Смена значения разлогинивает всех — это нормально при первой установке, но не меняйте его на живой системе без причины.
- **`GIT_PUSH_TOKEN`** — fine-grained GitHub PAT с доступом только к этому репозиторию, permission **Contents: Read and write**. Тот же класс токена, что сегодня лежит в секрете `GH_PAT` у `api-write.yml` — можно завести отдельный, не переиспользуйте секрет из Actions. Если оставить пустым, бэкенд поднимется и будет работать, но фоновый пуш-бэкап отключён (см. лог контейнера: `GIT_PUSH_TOKEN not set`). Можно завести позже — не блокирует первый запуск.
- **`DOMAIN`** — сейчас не используется (см. «Домен» ниже, порт вместо ACME) — Caddyfile жёстко слушает `:443`. Если/когда появится способ достучаться до реального 80 снаружи (выделенный IP, туннель), тогда DOMAIN и автоматический Let's Encrypt возвращаются — правки в Caddyfile на этот случай уже прокомментированы.

## 3. Поднять

```bash
docker compose up -d --build
docker compose logs -f backend   # проверить, что стартовал без ошибок
```

`docker compose ps` — оба сервиса `healthy`/`running`. Бэкенд слушает только внутри docker-сети (`expose`, не `ports`) — снаружи виден только Caddy, и только на `443` (см. «Домен» — `80` сейчас сознательно не проброшен).

## 4. Домен

**Обнаруженное ограничение (актуально для текущей VM, `95.190.201.194`):** это расшаренный IP провайдера — стандартные `80`/`443` снаружи отвечают не нашим Caddy, а чужим сервисом на этом же адресе (в нашем случае похоже на панель 1С). Реально наша машина видна только через проброшенные провайдером высокие порты (у нас `26502` → внутренний `443`, `26503` → `22` и т.д. — см. письмо от провайдера с конкретными номерами для вашей VM). Выделенный IP провайдер не даёт.

Следствия:
- Автоматический TLS у Caddy (Let's Encrypt, HTTP-01) не работает — challenge должен отвечать на настоящем 80 снаружи, а его нет. Вместо этого в Caddyfile `tls internal` — самоподписанный сертификат, браузер один раз предупредит «соединение не защищено», дальше работает нормально (можно добавить исключение или один раз поставить доверие сертификату на клиентских машинах, если хочется убрать предупреждение совсем).
- Адрес для входа — с портом явно: `https://<IP или домен>:26502`. Порт есть в URL всегда, обойти нельзя, пока не появится выделенный IP или туннель (Cloudflare Tunnel и т.п. — см. BACKEND_ARCHITECTURE.md, пока не настроено).
- DNS **можно** (необязательно, но удобнее IP) направить как раньше — A-запись `base.avtoscan42.ru` → `95.190.201.194` в ISPmanager — тогда адрес читается как `https://base.avtoscan42.ru:26502`, порт всё равно нужен.

Файл `CNAME` в репозитории трогать не нужно — он влияет только на то, какой custom-domain принимает GitHub Pages, а не на то, куда резолвится сам домен. Даже после переключения DNS на VM, вики на GitHub Pages остаётся доступна по дефолтному адресу:

```
https://keegooroomie.github.io/wikinest_avtoscan/
```

Это и есть URL аварийного фолбэка — не `base.avtoscan42.ru` (тот теперь указывает на VM), а дефолтный `github.io`-адрес. Стоит держать эту ссылку под рукой (закладка, канал в мессенджере) отдельно от основного домена.

## Аварийный режим (VM недоступна)

`deploy.yml` (публикация `index.html`/`style.css` на Pages) триггерится только пушем в сами эти файлы или вручную — правки контента через `docs/**` его не задевают, так что в обычной жизни он и не должен срабатывать сам по себе. Держите его явно выключенным между авариями:

- **Выключить** (после того как VM встала и стала основным путём): GitHub → Actions → `Deploy to GitHub Pages` → `•••` → *Disable workflow*.
- **Включить при аварии**: там же → *Enable workflow* → *Run workflow* (используется `workflow_dispatch` — не обязательно ждать пуша в `index.html`). После этого `base.avtoscan42.ru`... нет, домен уже указывает на упавшую VM — пользователям нужно раздать `https://keegooroomie.github.io/wikinest_avtoscan/` напрямую, пока не решите вернуть DNS на Pages вручную (если авария длинная) или поднять VM заново.

`build-tree.yml` и `api-write.yml` трогать не нужно — они должны оставаться включёнными всегда, это то, что вообще делает аварийный путь через `index.html` рабочим (запись через Actions, пересборка `tree.json` на стороне GitHub).

### Реконсиляция после аварийной правки

Если во время недоступности VM кто-то правил вики через `index.html` (аварийный путь):

1. Поднимая VM обратно, **до** того как бэкенд начнёт принимать новые правки: `git -C /srv/wikinest pull origin main`. Обычный (без `--force`) `git push` из фонового таймера бэкенда сам откажет, если этого не сделать — увидите ошибку в `docker compose logs backend` или в `/api/sync/status`, это ожидаемо, а не баг.
2. После `pull` — `docker compose restart backend`, чтобы пересобранный на GitHub `tree.json` подхватился (или дождаться следующей правки, она пересоберёт индекс сама).
3. Вернуть `base.avtoscan42.ru` на IP VM, если DNS временно переключали на Pages.
4. Выключить `deploy.yml` обратно.

## Бэкап / восстановление на новой VM

Диск текущей VM — не единственная копия (ежедневный `git push`, см. BACKEND_ARCHITECTURE.md), но если VM теряется между пушами, до `PUSH_INTERVAL_SECONDS` последних правок восстановить неоткуда — это осознанный компромисс, не баг. Проверять, что пуш реально идёт: `/api/sync/status` (или кнопка «Запушить сейчас» в Настройках `docs.html`) — таймер, который тихо перестал работать (протухший токен, сеть), ничем другим не выдаст себя.

Поднять на новой VM: шаги 1–4 выше — `git clone` (принесёт состояние на момент последнего успешного пуша), новый `.env` (тот же `GIT_PUSH_TOKEN`, можно сгенерировать новый `SECRET_KEY` — сессии просто разлогинятся, `roles.json` не затронут), `docker compose up -d --build`, перецепить DNS на новый IP.

## Обновление кода (`docs.html`, `backend/`, `style.css`, `Caddyfile`)

Это не про контент (тот пишется мгновенно бэкендом) — только когда меняется сам код приложения. Раз меняется редко, полноценный CI/CD не нужен:

```bash
cd /srv/wikinest
git pull origin main
docker compose up -d --build   # пересоберёт backend, если изменился в backend/
docker compose restart caddy   # подхватить новые docs.html/style.css/Caddyfile
```

**Оба рестарта обязательны, включая `caddy` — даже для статики.** `docs.html`, `style.css` и `Caddyfile` смонтированы в контейнер как bind-mount **отдельных файлов**, не директории (см. `docker-compose.yml`, сервис `caddy`). `git pull` не редактирует файл на месте — при fast-forward checkout git пересоздаёт файл (новый inode), а примонтированный в контейнер путь остаётся привязан к старому, пока контейнер не пересоздать/перезапустить. Без `restart` Caddy будет тихо отдавать старую версию — проверить можно по `Last-Modified`/`ETag` в заголовках ответа (`curl -I` на любой путь) или сравнением inode (`stat` на хосте и `docker compose exec caddy stat` внутри контейнера — разные числа значат старый маунт).

`docker compose up -d --build` **не** решает это само по себе — compose не видит в этом изменения конфигурации сервиса, только сам файл на диске поменялся, а значит нет причины пересоздавать контейнер.

**`tree.json`/`search.json` больше не подвержены этой проблеме.** Раньше они были такими же single-file bind-mount и ловили ровно этот баг на каждом `git pull`/merge (симптом: правки видимости через кнопку «глаз» или структура дерева выглядят так, будто не сохраняются — на самом деле сохранялось на диске, просто Caddy отдавал старую копию). Теперь Caddy проксирует `/tree.json` и `/search.json` на бэкенд (см. `Caddyfile`, `app.py` — `serve_tree_json`/`serve_search_json`), который читает их из своего собственного `/data`-маунта — тот же процесс, что их и пишет, инод-рассинхрона в принципе быть не может. Это касалось и обычного редактирования контента через приложение (`open(path,'w')` на месте, без смены инода — видно сразу и раньше), и особенно `git pull`/merge на самой ВМ (пересоздаёт файл) — второе теперь тоже не требует рестарта Caddy.
